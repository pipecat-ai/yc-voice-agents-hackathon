## Context
The Nemotron streaming STT client (`server/nvidia_stt.py`) shows cumulative transcripts in the RTVI UI: when a new turn starts, all prior turns' text appears in the live `user:` field and the new speech streams onto the end, then the per-turn final replaces it. The server keeps a warm WebSocket session whose decoder hypothesis persists across turns (no per-turn true-boundary reset), and it emits **interims as the full cumulative hypothesis** while emitting **finals as per-turn deltas**. We fix this client-side (no server edits) with: (1) strip the already-committed prefix from interim frames (best-effort cosmetic); (2) only stream speech-bounded audio to the server so inter-turn silence/echo isn't decoded into spurious tokens, with WS ping/pong keepalive. **A simpler alternative (sending `end` instead of `reset{finalize:true}` at VAD stop, which force-finalizes AND cold-resets per turn) is evaluated in Step 1 and may supersede Part 1 â€” see the decision checkpoint.**

### Part 1 (interim prefix-strip) correctness
- **CONDITIONAL on Step 1.** Enable prefix-stripping ONLY if Step 1 confirms the live server produces *continuous cumulative interims* (continuous mode). - **Best-effort cosmetic only.** It must never corrupt/drop user-visible final text. On ANY prefix mismatch, emit the interim unchanged and `logger.debug` the mismatch (short prefix sample) so divergence is observable.
- **Finals are DELTAS â†’ append-only.** Update the committed accumulator by APPENDING each hard `finalize:true` final's non-empty tokens (same branch that emits the `finalized=True` frame). Do NOT "reset/realign committed to a final frame" â€” a final is the per-turn delta (`server.py:7926`/`7955`, `8936`/`8968`), so resetting would drop all prior turns. Ignore soft (`finalize:false`) finals, empty finals, and interims for accumulation.

- The accumulator is **session-bound**: reset in `_connect_websocket`/`_disconnect_websocket` (reconnect-safe), never per turn.

### Accepted regressions / known limitations (document in code + verify in Step 6)
- **Transcription-fallback turn-start is weakened:** with gating, a user turn starts only when VAD fires (the TranscriptionUserTurnStartStrategy can't fire if no audio reaches STT). This couples turn-starting to VAD onset reliability. Accept for now; Step 6 must verify quiet/short utterances still trigger turns.
- **Reconnect spanning a speech segment** may lose that segment (base buffers audio while `_reconnecting` and replays after; our VAD stop may have already fired). Mitigation: reset segment state on reconnect; accept the rare loss. Do not attempt full control-event buffering in this iteration.

### Concurrency (narrow, accurate invariant)
- `InputAudioRawFrame` and VAD frames are **system frames**, processed immediately and serially within this processor (`frame_processor.py:999`), so `_user_speaking` / `_audio_ring` need no extra lock. VAD runs in the **downstream aggregator** and broadcasts `VADUserStartedSpeakingFrame` **UPSTREAM** to STT (`llm_response_universal.py:862`); by the time it reaches STT the onset audio has already passed through STT and been buffered in `_audio_ring`, so the flush-on-start captures it (pre-roll covers the round-trip).
- The WS **receive task** is a SEPARATE task. In steady state it mutates only `_committed_tokens` via `_handle_transcript`. **`cancel()` MUST cancel/await `_receive_task` BEFORE its manual `recv()` drain** â€” otherwise two coroutines call `recv()` on the same websocket (which raises) AND race on `_committed_tokens`. Guard WS sends with the existing `_audio_send_lock`.
- **Reconnect caveat (accepted):** auto-reconnect (`WebsocketSTTService._reconnect_websocket`) calls `_disconnect_websocket()`/`_connect_websocket()` from the receive task, and those reset the Part 2 session state (`_user_speaking`/`_audio_ring`/`_audio_bytes_sent`) too â€” so during a reconnect the receive task and the process task (audio/VAD frames) can both touch that state. asyncio is single-threaded (no torn writes), and the only observable effect is the already-accepted "reconnect spanning a speech segment may be lost." Not guarded further by design.
